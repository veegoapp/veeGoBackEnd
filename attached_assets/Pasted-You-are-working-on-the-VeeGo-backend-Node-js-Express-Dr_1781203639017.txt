## FIX 1 — Secure the exposed driver assignments endpoint

**Problem:**
`GET /api/shuttle/assignments` has no authentication guard. Anyone on the internet can call it and get all driver names, phone numbers, and bus assignments without logging in.

**What to do:**
- Add the `authenticate` JWT middleware to `GET /shuttle/assignments` in `artifacts/api-server/src/routes/shuttle.ts`
- The route should only be accessible to authenticated users (any role: user, driver, admin)
- Test: calling the endpoint without a token should return `401 Unauthorized`
- Test: calling it with a valid token should return data normally

---

## FIX 2 — Prevent wallet manipulation

**Problem:**
`POST /api/wallet/topup` currently accepts any positive amount and adds it directly to the user's wallet with no payment gateway. Any user can give themselves unlimited money.

**What to do:**
- Add a maximum top-up limit per request: 1000 EGP (store this limit in the `settings` table with key `wallet_max_topup` so it can be changed from the admin dashboard later)
- Add a daily top-up limit per user: 2000 EGP (store this in `settings` table with key `wallet_daily_topup_limit`)
- If the request exceeds either limit, return `400 Bad Request` with a clear Arabic + English error message
- Read the limits from the `settings` table (not hardcoded), with a fallback to 1000 and 2000 if the keys don't exist yet
- Add an admin endpoint `PATCH /api/admin/settings/wallet-limits` that allows updating both limits from the admin dashboard
- Add the two settings keys to the database seed/migration if not already present
- Test: top-up of 500 should succeed
- Test: top-up of 1500 should fail with clear error
- Test: two top-ups of 800 in same day should fail on the second one

---

## FIX 3 — Auto-activate driver account after admin approves all documents

**Problem:**
When an admin approves all required documents for a driver, the driver account status stays `pending` — the admin has to manually change it. This is a missing automation.

**What to do:**
- Required documents that must ALL be approved before auto-activation:
  `national_id_front`, `national_id_back`, `driving_license_front`, `driving_license_back`, `vehicle_license_front`, `vehicle_license_back`, `profile_photo`, `vehicle_photo`
- Criminal record (`criminal_record`) is NOT required for auto-activation (it becomes required after 30 trips)
- In `PATCH /api/driver-documents/:id` (the approve/reject endpoint): after every approval, check if ALL required documents for that driver are now approved
- If yes: automatically set `drivers.status = 'active'` and `users.isVerified = true` for that driver
- After auto-activation, create a notification for the driver: title `"Account Activated"` / `"تم تفعيل حسابك"`, body `"Your account has been approved. You can now start working."` / `"تمت الموافقة على حسابك. يمكنك الآن البدء في العمل."`
- Also emit a socket event `driver:account:activated` to the driver's personal socket room `driver:{userId}`
- Test: approve all required docs one by one → on the last approval, driver status should flip to active automatically
- Test: approving only some docs should NOT activate the account

## FIX 4 — Auto-refund passengers when admin cancels a trip

**Problem:**
`PATCH /api/trips/:id/cancel` and `DELETE /api/trips/:id` cancel or delete a trip but do not refund passengers who already have bookings on that trip.

**What to do:**
- In `PATCH /api/trips/:id/cancel` in `artifacts/api-server/src/routes/trips.ts`:
  - After marking the trip as cancelled, find all bookings for this trip that have status `confirmed` or `pending`
  - For each booking: refund the `totalPrice` back to the passenger's wallet
  - Create a `walletTransactions` record for each refund: type `refund`, description `"Trip cancelled by admin - refund"` / `"تم إلغاء الرحلة من قبل الإدارة - استرداد المبلغ"`
  - Update each booking status to `cancelled`
  - Create a notification for each affected passenger: `"Your trip has been cancelled and your money has been refunded"` / `"تم إلغاء رحلتك وتم استرداد المبلغ"`
- Apply the exact same refund logic to `DELETE /api/trips/:id`
- All of this must happen in a single database transaction (all or nothing — if any step fails, roll back everything)
- Test: create a trip with 3 bookings → cancel the trip → all 3 passengers should be refunded automatically
